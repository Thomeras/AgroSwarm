TERMINAL 6 — route provider
---------------------------------------
```bash
# Legacy obstacle_avoidance_mission is archived source-only and is no longer
# installed as a scout_control console script. Publish typed/JSON target_cmd
# messages from swarm_agent, Swarm Center, or a temporary external test node.
```

Rychly manualni smoke test targetu:
```bash
ros2 topic pub --once /drone_0/avoidance/target_cmd_json std_msgs/String \
  '{data: "{\"target_id\":\"test1\",\"command\":\"goto\",\"target_ned\":[3.0,0.0],\"altitude_m\":5.0}"}' \
  --qos-durability transient_local \
  --qos-reliability best_effort
```

POZOR:
- `altitude_m` je kladna vyska nad zemi, napr. `5.0`
- runtime ji prevadi na PX4 NED `z=-5.0`
- poslat `altitude_m:-5.0` je spatne a vysledny setpoint bude `z=+5.0`
  (tj. dron se snazi jit 5 m dolu do zeme misto vzletu)

Poznamka:
- pokud ladis konkretni beh, muzes obema nodum dat stejny `log_run_label:=...`
  a logy pak najdes v `logs/avoidance_logs/`
- `obstacle_detector` uz neni pro tento test potreba; jeho role je sloucena do
  `obstacle_avoidance_runtime`
- `obstacle_avoidance_runtime` je obecna per-drone node:
  - dela depth obstacle detection
  - dela scan / detour / PX4 offboard rizeni
  - prijima route cile zvenci pres `/drone_N/avoidance/target_cmd`
- archivovany `obstacle_avoidance_mission` uz neni instalovany jako entrypoint
- mise se po kazdem clearu vraci zpet na `home` pad a teprve potom leti na dalsi
  mission pad
- aktualni avoidance scan dela:
  - kratke zastaveni / hover
  - pomaly 360 stupnu spin scan na miste
  - ulozeni point cloudu a RGB reference snimku do `logs/avoidance_logs/*_assets/`
  - vyber objizdne strany z nasbirane lokalni mapy a drzeni teto strany v pameti
- jemne ladeni:
  - `scan_hover_ticks:=15` urcuje delku zastaveni pred spin scanem
  - `scan_spin_ticks:=80` urcuje delku pomaleho 360 scan otoceni
  - `scan_free_distance_m:=4.0` rika od jake clearance se sektor bere jako volny
  - `obstacle_memory_clear_ticks:=12` rika, po kolika clear tickech se smaze
    pamet na zvolenou stranu

CO CEKAT PRI TESTU
------------------
- `obstacle_avoidance_runtime` bere depth z `/drone_0/depth/image_raw`
- route provider musi publikovat cile na `/drone_0/avoidance/target_cmd`
- pri `warn` zpomaluje
- pri `critical` planuje bocni detour
- po projeti detour bodu pokracuje na puvodni target
- mezi jednotlivymi runy se vraci na `home` pad a dalsi route posila mise az potom

Testovaci cile:
- `North Wall`  -> `NED(22, 0)`
- `East Poles`  -> `NED(0, 22)`
- `NE Building` -> `NED(18, 18)`
- `NNW Fence`   -> `NED(22, -12)`
