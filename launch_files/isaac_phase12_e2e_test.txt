ISAAC PHASE 1+2 E2E VERIFICATION — MANUAL LAUNCH ORDER
======================================================

End-to-end overeni produkcniho stacku v Isaac Sim PRED zacatkem Phase 3.

Co overujeme:
- PHASE 1: obstacle_avoidance_runtime je jediny PX4 setpoint owner;
  swarm_agent bezi jako delegator (`navigation_backend=avoidance_runtime`);
  EKF / sensor health gates funguji; `/drone_N/avoidance/status` kontrakt.
- PHASE 2: pad metadata (pad_id, status, charging_capable, orientation_deg,
  service_priority); polygon boundary capture (B + F klavesy);
  polygon grid generation s `cell_class` per bunka; 4-corner legacy
  fallback; full setup state machine az do `READY_FOR_MISSION`.

Tohle je Isaac Sim varianta. Gazebo verzi viz `phase12_e2e_test.txt`.


DULEZITE
--------
- ROS2 backend spoustej AZ POTOM, co je Isaac Sim opravdu propojeny s PX4.
  Pokud PX4 stale vypisuje `Waiting for simulator to accept connection on
  TCP port 4560`, Isaac jeste neni korektne pripojeny.
- `colcon build` poustej z rootu workspace
  (`/home/tj/_Data/_Projekty/TJlabs/scout_ws`), ne z `~`. Jinak `colcon`

  zadny stary direct-control node (manual_controller v PX4 owner rezimu,
  offboard_control, obstacle_avoidance_mission) soucasne.
- `simulation_cam.py` pust jen JEDNOU za Isaac session (jinak duplikatni
  publishery na `/drone_0/camera/image_raw`).
- `camera_info` neni v Isaac/Pegasus pipeline garantovany. Pokud Phase 3

DOPORUCENY POSTUP PRO PHASE 2 OVERENI V ISAAC
---------------------------------------------
1. Po startu drone_0 pockej, az field_setup_coordinator hlasi `ASSIGN_PADS`.
2. Ve `field_setup_tool` (T6) nebo Swarm Center (T7) stiskni `1` ->
   assign pad_0 dronu_0.
3. Coordinator prejde do `CAPTURE_BOUNDARY`.
4. V Swarm Center tab `Manual` (nebo curses M-mode v T6) preleti drona po
   obvodu pole.
5. Na kazdem rohu pole stiskni `B` (boundary point).
6. Po zachyceni N polygon vertices stiskni `F` (close polygon).
7. Coordinator prejde do `GENERATE_GRID` -> `WAITING_FOR_LANDING`.
8. Po landed confirm prejde do `READY_FOR_MISSION`.
9. Stiskni `S` pro start mise.

KDYZ TO SPADNE — ISAAC SPECIFIC
-------------------------------
Pegasus / Isaac chyby:
- `new stage was opened`
- `World invalidated`
- `'World' object has no attribute '_scene'`

Postup:
1. zavrit vehicle load flow
2. znovu otevrit stage (`agro_field.usd`)
3. pockat na plne nacteni
4. az potom znovu PX4 backend + Load Vehicle + `simulation_cam.py` (1x) + Play

Pokud PX4 zustane viset na `Waiting for simulator to accept connection on
TCP port 4560`, ROS2 backend (TERMINAL 5) jeste NESPOUSTEJ.

Pokud `/drone_0/camera/image_raw` nebo `/drone_0/depth/image_raw` ma
publisher count > 1: `simulation_cam.py` byl spusten vicekrat.
Restartuj Isaac a pust ho jen jednou.

Pokud `/drone_0/depth/image_raw` netece, runtime nebude mit z ceho
pocitat lokalni mapu — avoidance bude na slepo.

Pokud PX4 hlasi `Ready for takeoff!`, `/drone_0/avoidance/status` ukazuje
`phase: TAKEOFF`, ale vehicle zustava stat:
- zkontroluj `/fmu/out/vehicle_status_v3`
- zkontroluj `/fmu/out/vehicle_control_mode`
- zkontroluj `/fmu/out/vehicle_command_ack_v1`
- zkontroluj `/fmu/out/failsafe_flags`

Rychla sada:
```bash
ros2 topic echo --once /drone_0/avoidance/status | head -20
ros2 topic echo --once /fmu/out/vehicle_status_v3 | grep -E '^(arming_state|nav_state|failsafe)'
ros2 topic echo --once /fmu/out/vehicle_control_mode | head -15
timeout 5 ros2 topic echo /fmu/out/vehicle_command_ack_v1
ros2 topic echo --once /fmu/out/failsafe_flags | grep -E 'local_position_invalid|offboard_control_signal_lost|battery_warning'
timeout 3 ros2 topic hz /fmu/in/trajectory_setpoint
timeout 3 ros2 topic hz /fmu/in/offboard_control_mode
```

Interpretace po dnesni session:
- `local_position_invalid: true` -> EKF jeste neni healthy, zkontroluj MAG/BAT params
- `local_position_invalid: false` + `phase: TAKEOFF` + `flag_armed: false`
  -> problem je v arm/offboard handoff, ne v target delivery
