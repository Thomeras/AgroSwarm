======================================================================
1. EXECUTIVE SUMMARY
======================================================================

Repo uz ma docela citelnou architekturu a v docs je pomerne jasne definovany cil:
- mission vrstva ma rozhodovat o tom "kam"
- runtime / avoidance vrstva ma rozhodovat o tom "jak se tam bezpecne dostat"
- UI, debug a operator tooling nema lezet v produkcni flight-control ceste

Nejvetsi realne nesoulady, ktere dnes v kodu vidim:

1. System je silne zavisly na `std_msgs/String` + JSON payloads.
   To je videt skoro vsude: `swarm_agent`, `swarm_coordinator`, `field_setup_coordinator`,
   `mission_launcher`, `home_manager`, `gcs_bridge`, `spray_controller`, `cell_data_recorder`,
   `ml_interface`, `obstacle_avoidance_runtime`, `obstacle_avoidance_mission`.
   Dlouhodobe to zhorsuje typovou bezpecnost, validaci a kompatibilitu QoS i verzovani payloadu.

2. `swarm_agent` stale neni cisty "mission executor / route provider", ale porad umi byt
   flight-control owner a ma vlastni avoidance / terrain-following logiku.
   To je v primem napeti s cilovou architekturou popsanou v `docs/plans/update_plans.txt`.
   Konkretne je to videt uz v hlavicce a publishers na PX4 v
   `src/scout_control/scout_control/swarm_agent.py:13-27` a `:219-232`.

3. `obstacle_avoidance_runtime` je produkcne spravny smer, ale stale je to velmi velky orchestrator.
   Ma 1504 radku, desitky parametru a drzi najednou state machine, PX4 offboard, perception,
   local mapping, planning, scan lifecycle i logging. Interni moduly uz vznikly, ale node samotny
   je stale "god object". Viz `src/scout_control/scout_control/obstacle_avoidance_runtime.py:149-191`.

4. Nektere nody maji spravnou roli, ale duplicitu odpovednosti:
   - `field_setup_coordinator` si grid generuje sam, prestoze existuje `grid_generator`
   - `manual_commander`, `manual_controller`, `field_commander`, `offboard_control`,
     `terrain_follower` a cast `swarm_agent` vsechny resi vlastni PX4 offboard loop
   - `home_manager` a `manual_controller` oba pracuji s RTH/pad assignment flow

5. Tooling/debug nody jsou zdrave jako reference, ale jsou ulozene ve stejnem runtime baliku
   jako produkcni core. To dnes neblokuje vyvoj, ale casem to ztezi launch composition,
   ownership i onboarding.

2. SYSTEMOVE SLABINY
======================================================================

A. Typed messages chybi tam, kde uz system prerostl experiment fazi
- Dnes to jeste funguje, ale mezivrstva mezi nodama je krehka.
- Chybi schema, validace, discovery, autocomplete i jasna migrace payloadu.
- Nejvic bych typoval:
  - `/swarm/drone_status`
  - `/swarm/task_status`
  - `/swarm/rth_request`
  - `/swarm/mission_ready`
  - `/{drone_ns}/avoidance/target_cmd`
  - `/{drone_ns}/avoidance/status`
  - `/swarm/pad_assignment`
  - `/field/setup_complete`

B. Prilis mnoho flight-control owneru
- Dnes umi primo publikovat do PX4 minimalne:
  `offboard_control`, `perimeter_flight`, `field_commander`, `manual_commander`,
  `manual_controller`, `terrain_follower`, `swarm_agent`, `obstacle_avoidance_runtime`.
- To je hodne na jeden workspace.
- Doporuceni: explicitne oznacit jen dva povolene production ownery:
  1. `obstacle_avoidance_runtime`
  2. pripadne minimalisticky fallback / test owner

C. Nekonzistentni lifecycle nodu
- `grid_generator` spousti hlavni praci primo v `__init__` (`grid_generator.py:42-64`).
- `mission_launcher` ukoncuje proces pres `raise SystemExit(0)` z timeru (`mission_launcher.py:151-156`).
- Nektere UI nody maji `curses` nebo `cv2.imshow` a vlastni thread model.
- To je rozumne u tooling nodu, ale je dobre to jasne oddelit od production path.

D. Hardcoded drone topology
- Nektere nody jsou uz parametrictejsi (`swarm_coordinator`, `field_setup_coordinator`, `gcs_bridge`).
- Jine porad predpokladaji konkretne `drone_0` a `drone_1`, napr. `cell_data_recorder.py:64-74`
  nebo workflow kolem `manual_controller`.
- Pokud ma roj rust, tohle zacne bolet.

E. Prilis mnoho domacich protokolu a topic conventions
- Misi, UI, GCS bridge a avoidance cast propojuji vlastni JSON protokoly a topic naming.
- Neni to spatne jako mezikrok, ale chce to zapsat jako oficialni kontrakty a postupne typovat.

3. NODE-BY-NODE HODNOCENI

Legenda:
- KEEP = role je v poradku, staci drobne upravy
- REFACTOR = role je spravna, ale implementace nebo hranice odpovednosti chce upravit
- TOOLING = uzitecne, ale nema byt v produkcni ceste
- DEPRECATE = drzet jen docasne / reference / test harness

1. `obstacle_avoidance_runtime` - REFACTOR, ale zustava production core
- Plus:
  - odpovida cilove architekture, mission target prijima zvenci
  - uz pouziva interni moduly `depth_projector`, `local_mapper`, `local_planner`, `scan_manager`
- Slabe:
  - je prilis velky a nese moc odpovednosti najednou
  - ma prilis mnoho parametru a stavu v jednom objektu
  - porad pouziva JSON prikazy a JSON status rozhrani
- Co upravit:
  - vytahnout command/status schema do typed msgs nebo aspon central parser layer
  - oddelit phase machine, PX4 publisher adapter a ROS IO adapter do samostatnych trid
  - zmenit node z "god object" na tenky orchestrator

2. `swarm_agent` - REFACTOR, architektonicky nejdulezitejsi dluh
- Plus:
  - jako mission executor dava smysl
  - umi frontu bunek, report progressu a vazbu na swarm vrstvu
- Slabe:
  - stale primo publikuje PX4 commandy (`swarm_agent.py:219-228`)
  - drzi vlastni avoidance, terrain-following a direct navigation logiku (`:109-124`, `:191-217`)
  - je prilis velky (1071 radku)
- Co upravit:
  - dotahnout ho do role route provideru nad `obstacle_avoidance_runtime`
  - nechat mu jen mission state, cell queue, reporting a high-level command generation
  - odstranit z nej direct PX4 owner path po stabilizaci runtime

3. `field_setup_coordinator` - REFACTOR, role spravna, hranice ne
- Plus:
  - jasny stavovy automat
  - dobra orchestrace setup workflow
- Slabe:
  - dela i generovani gridu, i kdyz na to existuje `grid_generator`
  - velka cast komunikace je String JSON
- Co upravit:
  - delegovat grid generation do `grid_generator` nebo sdilene service/library
  - ponechat si jen setup orchestration a readiness gating

4. `grid_generator` - KEEP/REFACTOR
- Plus:

- Co upravit:

8. UPDATE 2026-04-22 - SWARM_AGENT / RUNTIME REFACTOR STATUS
======================================================================

Od puvodniho auditu uz probehl dulezity refaktor kolem `swarm_agent` a
`obstacle_avoidance_runtime`.

Aktualni stav:

1. `swarm_agent` je prakticky prepnuty na runtime-first model
- default `navigation_backend=avoidance_runtime`
- v runtime backendu se nespousti direct PX4 ownership vetev
- `swarm_agent` posila high-level commandy do `/{drone}/avoidance/target_cmd`
- dokoncenou bunku odvozuje z runtime feedbacku, konkretne z `last_completed_target_id`

2. `obstacle_avoidance_runtime` je posilen jako execution owner
- robustnejsi ingestion commandu
- lepsi tolerance na aliasy command / envelope / target payloadu
- nove additive eventy:
  - `command_accepted`
  - `command_rejected`
- rozsiren `avoidance/status` o additive mission-feedback pole:
  - owner
  - execution state
  - feedback state

3. Dokumentace byla prubezne synchronizovana
- `CLAUDE.md` aktualizovan podle runtime-first modelu
- `codex.md` aktualizovan podle noveho rozdeleni odpovednosti

Co je timto uz architektonicky splneno:
- `swarm_agent` uz neni hlavni preferovany flight-control owner
- `obstacle_avoidance_runtime` je pro tuto execution path povazovan za produkcniho ownera
- hlavni zlom ve smeru "mission provider -> runtime executor" uz probehl

Co je zatim prechodove:
- `navigation_backend=direct` stale existuje jako fallback kompatibilita
- stara direct-control logika je v `swarm_agent.py` jeste fyzicky pritomna
- pro runtime backend je ale odpojena a neni primarni cestou

Co jeste zbyva do uplneho dokonceni:
1. odstranit legacy `direct` backend ze `swarm_agent`
2. udelat launch/config cleanup, aby `avoidance_runtime` byl konzistentni default ve vsech scenarich
3. potvrdit full E2E swarm mission flow pred definitivnim smazanim stare direct vetve

Overeni po integraci:
- `pytest`: 23 passed
- `py_compile` pro `swarm_agent.py` a `obstacle_avoidance_runtime.py` proslo

Dopad na puvodni audit:
- puvodni hodnoceni `swarm_agent` jako nejdulezitejsiho architektonickeho dluhu bylo spravne
- nejvetsi cast tohoto dluhu uz byla zmensena
- dalsi krok uz neni "velky redesign", ale dokonceni cleanupu a odstraneni legacy fallback cesty
