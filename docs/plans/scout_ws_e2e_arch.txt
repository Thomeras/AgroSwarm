SCOUT_WS – End-to-End Agrodrone Swarm Architecture
Field setup, mapping, mission planning, swarm execution, obstacle avoidance, spraying, monitoring, charging, reporting
Context: ROS 2 Jazzy, PX4, Isaac Sim / Pegasus, Python runtime, Swarm Center GCS
Design goal: mission-first field operations with pre-mission world modeling and local onboard safety autonomy

1. Executive summary
The system is intentionally split into two planning horizons. Before an operational mission, the field is bounded, mapped and converted into a field model containing perimeter, terrain height, static obstacles, no-go zones and mission constraints. During an operational mission, each drone follows a preplanned task assignment while onboard obstacle avoidance remains active as a local safety and recovery layer.
The single flight owner for any autonomy-enabled mission is obstacle_avoidance_runtime. Mission logic, swarm allocation and operator tools do not publish competing PX4 setpoints; they delegate high-level targets such as goto, hold, return_home or land. This preserves a clean authority boundary between mission intent and safe execution.
The online obstacle layer is depth-camera-first. The drone maintains a rolling 2.5D local map around itself, uses warn-stage drift for mild avoidance, and escalates to hover plus slow 360° scan when the local corridor becomes critical or repeated replanning fails. Scan data persists longer than ordinary live depth observations and is used to enrich the local map before short-horizon replanning.
Swarm Center is not only a live dashboard but also the planning surface for the entire system. It stores field boundaries, mapping outputs, static obstacle and terrain layers, refined grid assignments, drone health, mission state, service history and agronomic reports. The objective is to make obstacle avoidance the exception rather than the default way of navigating around known field structures.
2. Design principles
Mission-first: Grid / waypoint mission planning remains the main navigation concept. Local avoidance only protects execution and handles uncertainty.
Planning before execution: Known terrain and known obstacles are integrated into mission planning before the main operation starts.
Single flight owner: Only obstacle_avoidance_runtime owns offboard flight setpoints during autonomous execution.
Swarm simplicity: Swarm logic assigns work, handles blocked drones and coordinates pads; it does not run a heavy centralized path planner.
Source-agnostic mapping: The local mapper accepts point batches from live depth, scan clouds, peer tracks and later lidar or static mission layers.
Degrade gracefully: Warn drift, stop-hover, scan, local replan, blocked, return-home and abort are explicit steps, not ad-hoc exceptions.
Operational traceability: Every important event leaves structured logs: state transitions, blocked episodes, scans, chosen paths, spray actions and cell results.
3. End-to-end operational lifecycle
3.1 System startup and readiness
Boot PX4/ROS 2 stack, verify telemetry, camera, depth and any downward ranging sources.
Load home pad registry, field profiles, mission templates and agronomic task presets in Swarm Center.
Confirm drone inventory: battery state, payload state, sensor status, communication link and readiness class.
3.2 Home pad setup
Use manual setup tools to register each physical home pad and bind it to a drone or a pad pool.
Persist home_positions.json and expose home targets through home_manager so RTH remains deterministic.
Pads should carry metadata such as pad ID, NED coordinates, charging capability, occupancy state and service priority.
3.3 Manual field boundary capture
Use one drone in manual boundary mode to fly the outer perimeter and mark boundary points.
Store the boundary as a closed polygon in local NED plus optional GPS origin metadata.
Apply a safety inset buffer so operational paths do not scrape the exact field edge or fence line.
3.4 Initial grid generation
Generate a coarse operational grid from the field polygon.
Mark cells as inside, outside, edge, restricted or transitional.
This first grid is only a geometric base and is not yet terrain- or obstacle-aware.
3.5 Mapping mission
Run a dedicated mapping flight before the production task. This mission collects terrain height evidence, static obstacles, canopy structure and any permanent no-fly features.
Mapping flight should use obstacle avoidance as an active safety layer, but the priority is coverage and model quality rather than payload service.
Typical mapping outputs include height rasters, obstacle clouds, obstacle polygons, static no-go masks, confidence maps and coverage completeness flags.
3.6 Field model building
Fuse mapping results into a field model maintained by Swarm Center: perimeter, terrain layer, static obstacle layer, caution zones and operational corridors.
Generate refined no-go zones around permanent obstacles and safety buffers around fences, poles, buildings, irrigation equipment or trees.
Derive altitude recommendations or slope constraints where terrain relief changes mission execution risk.
3.7 Mission refinement and swarm allocation
Refine the coarse grid into a mission grid that respects static obstacles, no-go zones and terrain constraints.
Split work for any number of drones by availability, payload, battery, capability and distance to target sectors.
Produce a mission package per drone: ordered cells or corridors, service type, target altitude mode, payload instructions and charging expectations.
3.8 Live execution
During live execution, swarm_agent or another route provider sends only high-level targets to obstacle_avoidance_runtime.
Runtime locally manages warn drift, critical scan, local replanning and PX4 setpoint ownership.
Unexpected obstacles, moving machinery, humans, animals or peer conflicts are handled onboard without invalidating the higher-level mission structure.
3.9 Charging, landing, pause and continuation
When a drone completes a block, reaches low battery, or loses payload viability, mission logic routes it to a suitable home/charge pad.
After charging or payload refill, Swarm Center can resume from the next pending task set rather than restarting the whole mission.
The same workflow supports manual pause, weather hold, operator override and safe continuation.
3.10 Reporting and agronomic outputs
At mission end, Swarm Center compiles field health, completed coverage, anomalies, pest indicators, spray history, blocked zones, charging history and unresolved cells.
The platform should support agronomic recommendations such as next fertilizer pass, priority inspection areas, repeat-mapping sectors or potential crop stress interventions.
4. Architecture layers
Operator / GCS layer: Swarm Center UI, status dashboards, map editing, mission planning, review, manual approvals and report generation.
Field model layer: Boundary polygon, grid model, terrain model, static obstacle model, no-go zones, cautions and mission constraints.
Swarm orchestration layer: Task allocation, cell assignment, drone availability, pad coordination, deferred work and mission progression.
Mission delegation layer: swarm_agent and other route providers convert assigned work into target_cmd messages without touching PX4 setpoints.
Drone runtime safety layer: obstacle_avoidance_runtime with internal local_mapper, local_planner, scan_manager and peer tracking.
Vehicle control layer: PX4 offboard interface, arming, takeoff, setpoint tracking, failsafe transitions and landing.
Payload / agronomy layer: spraying, growth monitoring, pest monitoring, cell recording, classification and recommendations.

6. Field setup and world modeling
6.1 Home pads and readiness
Each home pad should have a stable ID, NED position, optional GPS anchor, occupancy state, charging state and allowed drone classes.
Home setup is not only for takeoff/landing. It is also a scheduling input for battery rotation, payload refill and emergency return-home behavior.
6.2 Manual boundarying
The first operational version should use manual boundarying because it is reliable and easy to validate. Fly one drone manually or semi-manually around the field edge and store polygon points.
Boundary capture should support both sparse corner capture and denser sampled segments. Sparse mode is useful for simple rectangular plots; dense mode captures irregular boundaries, roads or cutouts.
Recommended file products: field_boundary.json, field_boundary_preview.png and a revision log with operator name and timestamp.
6.3 Mapping mission products
The mapping mission should create a field model rather than just a raw point cloud dump. That field model becomes the bridge between geometry and operations.
Recommended outputs: static obstacle polygons, obstacle confidence raster, terrain/height raster, traversability mask, canopy-density or crop-height proxy, and mapping coverage completeness.
Known static objects should be tagged and versioned so the planning layer can distinguish permanent infrastructure from uncertain temporary detections.
6.4 Refined grid and no-go zones
After mapping, generate no-go zones and caution buffers around static obstacles. Inflate them according to platform size, GPS / odometry uncertainty and service maneuver requirements.
Refined grid generation should cut or split cells that intersect no-go zones. Some cells may become partial-service cells, corridor cells or observation-only cells.
Terrain relief can be represented as both a continuous height model and simple operational classes such as flat, sloped, steep, low-clearance or poor-visibility zone.
7. Onboard obstacle avoidance and local autonomy
7.1 Core runtime rule
obstacle_avoidance_runtime is the only autonomous flight owner. All other nodes delegate mission intent and consume status; they do not issue competing PX4 setpoints.
7.2 Local mapping model
Use a rolling 2.5D local map around the drone, centered on the current pose. A practical default is 40 m × 40 m at 0.25 m resolution, with local occupancy, inflation, blocked-history and peer-cost layers.
Live depth observations have short persistence; scan-enriched observations persist longer. Unknown space remains unknown and should not be silently treated as free.
The local mapper should also accept mission-provided static context slices from Swarm Center, so onboard planning is not limited to the camera's instantaneous field of view.
7.3 Warn and critical behavior
Warn: reduce speed and drift using the best local corridor from the planner. The purpose is to resolve mild conflicts without stopping the mission every time.
Critical: stop horizontal motion, stabilize hover, perform a slow 360° scan, enrich the local map and trigger replanning.
Repeated failures: escalate to BLOCKED, recommend task deferral or return-home, and preserve all artifacts for review.
7.4 Field of view limitation and why global context matters
A forward-looking depth camera sees a frustum, not a full sphere around the drone. That means runtime local knowledge is incomplete by design.
This limitation is addressed through three mechanisms: rolling memory, explicit 360° scan on demand, and static mission context downloaded from Swarm Center for the drone's current neighborhood.
As a result, the drone does not need perfect omnidirectional sensing at all times to operate safely in a mapped field.
8. Sending map context from Swarm Center back to drones
Swarm Center should not stream the entire field model at high rate to every drone. Instead, it should provide a compact local mission context slice around the drone's current region or current assigned sector.
This slice can include known static obstacle masks, no-go polygons, terrain penalties and preferred corridors. The onboard local mapper merges it as a lower-frequency static layer beneath live depth and scan data.
This approach closes the loop: mapping creates the field model, planning uses it to refine missions, and execution uses local extracts of the same model to reduce unnecessary online improvisation.
9. Mission types
Spraying / fertilization mission
Cells carry service prescriptions such as target dose, material type, altitude preference, revisit need and exclusion buffers.
Spray controller logs each action with location, cell ID, timestamp and dose. If a cell is only partially serviced due to obstacle or battery constraints, the remainder stays in the mission queue.
Growth monitoring mission
The primary payload is imaging or sensor evidence. Cell completion stores imagery, pose, time, height and derived crop indicators.
Swarm Center aggregates cell-level evidence into field-level health maps and trend reports.
Pest monitoring mission
This mission prioritizes anomaly localization, not just broad coverage. It may route drones to re-check suspicious cells with higher image overlap or slower speed.
Detected pest hotspots become follow-up targets for repeat monitoring or targeted spraying missions.
Mapping refresh mission
A shorter refresh mapping flight updates changes in terrain, structures or temporary equipment without rebuilding the entire field model from zero.
10. Swarm scaling for any number of drones
Allocation should be capability-aware, not only count-aware. Drones may differ by battery, payload, camera, sprayer capacity or maintenance state.
A practical strategy is sector-first allocation: split the refined field model into contiguous service sectors, then allocate cells or corridors within each sector to available drones.
When a drone becomes hard-blocked, low-battery or unavailable, unfinished cells are returned as deferred work. The coordinator then reassigns them to another drone or pushes them to the next sortie.
Peer drones are treated onboard as dynamic safety zones, but swarm-level planning should also reduce crossing trajectories and pad contention before launch.
11. Swarm Center UI requirements
Planning views
2D field view with perimeter, pads, refined grid, assigned sectors, no-go zones and active drone positions.
3D or height-aware field view showing terrain surface, static obstacles and optionally canopy / crop structure.
Boundary editor and no-go zone editor with revision history.
Execution views
Live mission state per drone: target, runtime state, blocked severity, battery, payload, charging, pad assignment.
Local avoidance overlays on demand: current subgoal, planned path, local grid snapshot, scan events, peer conflicts.
Operator actions: pause, hold, RTH, land, retry blocked task, approve deferred-cell reassignment.
Analysis and reporting views
Coverage completeness, serviced vs pending cells, spray history, blocked hotspots and repeated anomaly sectors.
Field health reports: growth condition layers, pest markers, revisit recommendations and suggested next treatment zones.
Maintenance and utilization reports: flight time, charge cycles, payload consumption, blocked-time and pad usage.

13. Charging, landing and continuity
Charging should be modeled as part of mission planning, not as an afterthought. The planner should know which pads can charge, refill or only land.
Mission execution needs resumability. A drone that lands to charge should re-enter the mission with the next valid pending work package rather than forcing global restart.
Return-home is not a separate simple mode outside avoidance. It is another high-level target that still passes through the same local safety runtime.
14. Reporting and agronomic recommendations
The platform should produce two report classes: operational reports and agronomic reports.
Operational reports cover sorties, blocked events, flight efficiency, pad usage, charging, coverage achieved, skipped cells and mission completion quality.
Agronomic reports cover crop growth indicators, anomaly hotspots, pest likelihood regions, spray history, untreated patches and recommended follow-up actions.
Recommendation examples: prioritize re-spray in cells with low application confidence, schedule a targeted pest validation pass in cells with repeated anomaly signatures, increase monitoring frequency in sectors with rapid growth divergence, and delay treatment in waterlogged or low-confidence terrain sectors.
15. Recommended implementation roadmap
Phase 1 – Stable onboard runtime: Finish the runtime split into local_mapper, local_planner, scan_manager and peer tracks. Keep depth-camera-first local avoidance stable and make swarm_agent a pure delegator.
Phase 2 – Boundary to base grid workflow: Finalize manual boundary capture, home pad registration and clean field_grid generation from polygons.
Phase 3 – Mapping mission pipeline: Create the mapping mission, field model outputs, obstacle extraction and terrain map generation.
Phase 4 – Refined grid and mission package generation: Use the field model to create no-go zones, terrain-aware grid refinement and drone-specific mission packages.
Phase 5 – Swarm Center planning and reporting: Expose the field model and planning controls in GCS, then generate execution dashboards and post-mission reports.
Phase 6 – Context slices back to drones: Send static local map extracts to drones by region so onboard local planning benefits from the global field model.
Phase 7 – Agronomic intelligence: Replace placeholders with real models for growth, pest detection, treatment recommendation and confidence estimation.
Phase 8 – Sensor and infrastructure expansion: Optionally add lidar, richer terrain sensing, automated refill workflow and more advanced service scheduling.
16. Final target state
The final system is a field operations platform, not only a flight controller. It begins with setup and mapping, transforms that into a mission-aware world model, allocates work across any number of drones, executes safely with onboard local autonomy and ends with actionable agronomic and operational reports.
Known obstacles should be avoided by design during planning; unexpected obstacles should be handled onboard during execution. This distinction is what keeps the system robust without overengineering it into a heavy urban-autonomy stack.
