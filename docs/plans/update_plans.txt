1. EXECUTIVE SUMMARY
======================================================================

Cilova architektura ma zustat mission-first:
- grid / waypoint mission zustava hlavni navigace
- obstacle avoidance zustava lokalni safety vrstva
- scan je fallback pro map enrichment + local replan, ne hlavni planner
- swarm zustava jednoduchy: separace, docasna blokace, pripadny reassign

Nejdulezitejsi rozhodnuti:
- `obstacle_avoidance_runtime` zustane produkcni per-drone runtime node
- ale jeho vnitrni logika se rozdeli na moduly `local_mapper`, `local_planner`, `scan_manager`
- tyto casti doporucuji nejdriv implementovat jako interni moduly/tridy ve stejnem procesu
- neni vhodne z nich ted delat dalsi ROS nody, protoze by to pridalo latenci, QoS problĂ©my a zbytecnou slozitost

Produkci cil pro kazdy dron:
- mission / swarm vrstva posila high-level target do runtime
- runtime drzi PX4 offboard, lokalni mapu, replanning, scan, safety reakce
- mission vrstva dostava zpet pouze stav, progres, blocked a completion

Prakticka migrace:
- kratkodobe zachovat dnesni topic `/{drone_ns}/avoidance/target_cmd` a `/{drone_ns}/avoidance/status`
- v prvni iteraci pouzit stale `std_msgs/String` JSON kvuli kompatibilite
- az po stabilizaci zavest volitelny `scout_msgs` package s typovanymi zpravam

4. C. RUNTIME STATE MACHINE [DONE 2026-04-22]
======================================================================

4.1 `IDLE`

Vstup:
- runtime start
- target dokoncen
- cancel bez dalsiho targetu

Akce:
- dron drzi pozici / bez aktivniho mission targetu
- mapper muze stale bezet a drzet kratkou memory
- status = ready/not-ready

Prechody:
- na `TAKEOFF`, kdyz prijde `goto` / `return_home` a je treba ziskat letovou vysku
- na `CRUISE_TO_TARGET`, kdyz uz je dron ve spravne vysce a target je aktivni
- na `LANDING`, kdyz prijde `land`
- na `ABORT`, pri kriticke chybe runtime

4.2 `TAKEOFF`

Vstup:
- aktivni target a dron neni v cruise altitude band

Akce:
- arm + offboard
- vystoupani na mission altitude / AGL target
- minimalni horizontal movement

Prechody:
- na `CRUISE_TO_TARGET`, kdyz altitude reached
- na `ABORT`, pri nav timeout / PX4 failure

4.3 `CRUISE_TO_TARGET`

Vstup:
- aktivni mission target
- planner ma direct nebo lehce korigovanou trasu

Akce:
- jet po direct line nebo planner subgoal
- planner refresh typicky 2 Hz
- mapper update typicky 10 Hz
- publish progress

Prechody:
- na `WARN_DRIFT`, kdyz obstacle je v warn zone ale direct path jeste neni hard-blocked
- na `STOP_HOVER`, kdyz critical obstacle pred nosem nebo planner nema direct corridor
- na `RETURN_HOME`, pri external RTH
- na `LANDING`, pri external land
- na `IDLE`, kdyz target complete

4.4 `WARN_DRIFT`

Vstup:
- obstacle vepredu v warn distance
- planner nasel mirny lateral bias bez potreby zastavit

Akce:
- snizit speed
- drzet mission heading, ale s lateral bias
- nepublikovat blocked, jen avoidance_active

Prechody:
- zpet na `CRUISE_TO_TARGET`, kdyz corridor zase cisty
- na `STOP_HOVER`, kdyz se free corridor uzavre nebo obstacle critical

4.5 `STOP_HOVER`

Vstup:
- critical obstacle
- repeated warn without safe subgoal
- detour failure

Akce:
- zastavit horizontal motion
- stabilizovat yaw / hover
- freeze current mission progress

Prechody:
- na `SCAN_360`, kdyz je potreba map enrichment
- na `LOCAL_REPLAN`, kdyz mapa uz ma dost dat a neni nutny scan
- na `BLOCKED`, kdyz hover timeout a stale neni plan

4.6 `SCAN_360`

Vstup:
- `STOP_HOVER` + duvod `critical`, `repeated_blocked`, `no_path_after_warn`

Akce:
- pomaly 360 yaw spin
- ukladat RGB + point cloud + meta
- importovat scan cloud do mapperu

Prechody:
- na `LOCAL_REPLAN`, po `scan_complete`
- na `BLOCKED`, pokud scan selze nebo planner i po scanu nema path

4.7 `LOCAL_REPLAN`

Vstup:
- scan complete
- planner trigger pri blockaci
- periodicky refresh pri detour execution

Akce:
- spustit A* na planning gridu
- vybrat subgoal na kratky horizont
- penalizovat problem zones a peer drony

Prechody:
- na `DETOUR_EXECUTION`, kdyz plan uspesny
- na `CRUISE_TO_TARGET`, kdyz je direct corridor zase volny
- na `BLOCKED`, kdyz `NO_PATH`

4.8 `DETOUR_EXECUTION`

Vstup:
- planner vratil lokalni path/subgoal

Akce:
- sledovat prvni cast lokalni path
- periodicky replannovat
- drzet mission target jako finalni referenci, neprepisovat ho detour waypointem natrvalo

Prechody:
- na `CRUISE_TO_TARGET`, kdyz se vrati direct corridor
- na `STOP_HOVER`, kdyz detour corridor zhori / obstacle critical
- na `BLOCKED`, po vice neuspesnych replan pokusech

4.9 `BLOCKED`

Vstup:
- planner `NO_PATH`
- scan + replan neuspesny opakovane
- blocked timeout prekrocen

Akce:
- hover
- publikovat `blocked=true`, `reassign_recommended=true`
- log event + planner/grid snapshot

Prechody:
- na `LOCAL_REPLAN`, pokud se mapa zmeni nebo operator/mission da retry
- na `RETURN_HOME`, pokud mission vrstva rozhodne o stazeni
- na `ABORT`, pri safety escalation

4.10 `RETURN_HOME`

Vstup:
- explicitni command
- mission-level abort / weather / battery / manual intervention

Akce:
- home je jen dalsi high-level target
- avoidance stale aktivni i pri RTH

Prechody:
- na `CRUISE_TO_TARGET`, pokud home neni blokovany
- na `STOP_HOVER` / `SCAN_360`, kdyz i RTH corridor je blokovany
- na `LANDING`, po dosazeni home

4.11 `LANDING`

Vstup:
- explicitni land command
- return_home complete

Akce:
- handoff na PX4 auto land nebo controlled descend

Prechody:
- na `IDLE`, po potvrzenem landing complete
- na `ABORT`, kdyz land flow selze

4.12 `ABORT`

Vstup:
- internal exception
- PX4 command failure
- repeated unrecoverable blocked + no safe home path

Akce:
- konzervativni hold / emergency procedure
- status `aborted`
- log maximalni kontext

Prechody:
- jen operator / external recovery

5. D. LOKALNI MAPA
======================================================================

5.1 Doporuceny model

Pouzit rolling 2.5D grid:
- horizontalni planner zustava 2D
- ale do bunek se uklada i jednoducha vyskovĂˇ evidence
- to staci pro field use case a neotevira full 3D urban stack

5.2 Doporucene parametry

Default:
- okno: 40 m x 40 m kolem dronu
- planner aktivni horizon: 12 m az 15 m dopredu
- resolution: 0.25 m
- grid size: 160 x 160

Proc:
- 0.25 m je dost jemne pro sloupky, stromy, hrany staveb a farm prekazky
- 160 x 160 bunek je porad levne pro Python A* v kratkem horizontu
- 40 m okno je dost pro scan-based local replanning, ale neni to pseudo-global map

5.3 Doporucena data na bunku

`LocalCell`:
- `occupancy_logodds: float`
- `last_observed_ts: float`
- `last_occupied_ts: float`
- `max_height_rel_m: float`
- `min_height_rel_m: float`
- `source_flags: int`
- `blocked_score: float`
- `visit_penalty: float`
- `peer_cost: float`

`source_flags`:
- bit 1 = live depth
- bit 2 = scan cloud
- bit 3 = peer drone mask
- bit 4 = blocked memory

5.4 Projekce depth dat do mapy

Pipeline:
1. depth frame -> `depth_projector`
2. stride downsample pro runtime mode
3. points transformovat do local NED/world XY
4. odfiltrovat range mimo napr. 0.4 m az 15 m
5. odfiltrovat body mimo collision band kolem vysky dronu
6. XY body zapsat do bunek jako occupied evidence

Collision band pro field drone:
- default relativni vertikalni band vuci dronu: `[-2.0 m, +2.5 m]`
- planner tedy ignoruje nepodstatne body hluboko pod dronem
- zaroven lze zachytit steny, tyce, stroje, stromy

5.5 Jak promitat scan cloud

Scan cloud:
- pouzit stejnou projekcni cestu jako live depth
- ale dat mu vyssi confidence a delsi decay
- scan body zapisovat bez agresivniho downsample

Doporuceni:
- live depth = rychla, kratka pamet
- scan cloud = pomalejsi, kvalitnejsi memory

5.6 Decay / zapominani

Pouzit 2 vrstvy decay:

Fast decay pro live depth:
- occupancy decay half-life cca 8 s az 12 s
- vhodne pro listi, noise, zmenu perspektivy

Slow decay pro scan/static memory:
- half-life cca 45 s az 90 s
- vhodne pro delsi prekazky, steny, kontejnery, stromy

Blocked-memory decay:
- 120 s nebo do zmeny mission target corridor

5.7 Problemove oblasti / repeated blocked

Evidovat zvlast:
- `blocked_score += 1.0` pri `BLOCKED`
- `blocked_score += 0.5` pri `detour_failed`
- planner ma tuto vrstvu jako cost, ne hard no-go

Smysl:
- stejne misto nebude planner zkouset znovu a znovu
- nevytvori se ping-pong mezi levou a pravou variantou

5.8 Ostatni drony v mape

Promitnout je jako dynamicke cost/disc zony:
- hard core radius: 3.0 m
- soft shell radius: 6.0 m
- predikce jen kratka, typicky 2 s dopredu
- pokud nejsou velocity data, pouzit posledni pozici + vetsi shell

Dulezite:
- ostatni drony nemaji jit do occupancy memory jako "staticka prekazka"
- patri do separatni dynamicke vrstvy s rychlym decay

5.9 Planning grid

Planner ma dostat od mapperu finalni 2D planning grid:
- occupied hard cells
- inflated obstacle cells
- peer no-go mask
- blocked history cost
- unknown/low confidence cost

Inflace:
- hard obstacle inflation: 1.0 m az 1.5 m
- peer drone inflation: min 3.0 m
- near-obstacle soft cost gradient: do 2.0 m navic

6. E. PLANNER
======================================================================

6.1 Co doporucuji

Pouzit kratkohorizontovy grid A* planner s vyberem subgoalu.

Proc:
- je dost robustni pro rolling 2D grid
- umi respektovat inflation + peer cost + blocked history
- je jednodussi a transparentnejsi nez MPC/TEB/voxel planner
- je adekvatni pro field drone, kde avoidance neni hlavni navigace

6.2 Co nedoporucuji

Nedoporucuji ted:
- full global planner pres cele pole
- 3D voxel planner jako produkcni default
- full SLAM mapu farmy
- centralized multi-agent path planner

Proc:
- neprinese adekvatni hodnotu pro otevrene pole
- zvedne integracni slozitost, tuning i failure modes

6.3 Planner flow

Pri kazdem plan requestu:
1. zkontrolovat direct line-of-sight k mission targetu nebo k projected horizon point
2. pokud volno, vratit `DIRECT`
3. jinak vybrat subgoal v lookahead vzdalenosti po smeru targetu
4. spustit A* na planning gridu k tomuto subgoalu
5. pokud A* selze, zkusit ring candidates vlevo/vpravo od mission smeru
6. pokud nic nevyjde, vratit `NO_PATH`

6.4 Jak vybirat subgoal

Prvni volba:
- final mission target, pokud je uvnitr okna a line-of-sight je bezpecna

Druha volba:
- projected target na mission bearing ve vzdalenosti 10 m az 12 m

Treti volba:
- kandidati na kruznici 8 m az 12 m kolem dronu
- preferovat male odchylky od mission bearing
- preferovat kandidaty s nizsim blocked cost a sirsim clearance

6.5 Costy v A*

Doporucene slozky costu:
- zakladni distance cost
- obstacle inflation cost
- blocked history cost
- peer drone dynamic cost
- heading deviation cost vuci mission smeru
- turn smoothness cost
- unknown-space penalty mensi nez hard obstacle, ale nenulova

Prakticky:
- obstacle cells = hard forbidden
- inflation = +5 az +25
- blocked history = +10 za kazdou uroven
- peer soft shell = +20 az +60
- heading deviation = mala penalizace, napr. +0.5 az +3

6.6 Planner vystup

`PlanResult`:
- `mode`: `DIRECT | SUBGOAL_PATH | HOLD | NO_PATH`
- `subgoal_ned: [x, y] | null`
- `path_ned: [[x1, y1], ...]`
- `path_cost: float`
- `clearance_m: float`
- `reason: str`
- `blocked_zone_ids: []`

6.7 Kdy planner spoustet

Planner trigger:
- 2 Hz v `CRUISE_TO_TARGET`
- okamzite pri `critical obstacle`
- okamzite po `scan_complete`
- znovu pri kazde zmene peer-drone no-go zony, ktera protina aktualni corridor

6.8 Kdyz neni cesta

Strategie:
- prvni `NO_PATH` -> `STOP_HOVER`
- pokud je mapa chuda -> `SCAN_360`
- pokud scan probehl a stale `NO_PATH` -> `BLOCKED`
- po 2 az 3 neuspesnych scan/replan cyklech -> `reassign_recommended`

6.9 Repeated blocked behavior

Doporucene limity:
- `soft_blocked_timeout_s = 15`
- `hard_blocked_timeout_s = 30`
- `max_scan_retries = 2`

Po prekroceni:
- runtime publikuje `blocked_severity = HARD`
- mission/swarm vrstva muze defernout aktualni cell / waypoint

6.10 Jak planner navazuje na scan

Scan nema vybrat "left/right" finalne.

Spravny flow:
- scan naplni mapper hustsimi body
- mapper vyrobi planning grid
- planner teprve zvoli subgoal/path

Toto je hlavni architektonicka zmena oproti dnesni heuristice.

7.2 Runtime -> mission

Topic:
- `/{drone_ns}/avoidance/status`
Publisher:
- `obstacle_avoidance_runtime`
Subscriber:
- `swarm_agent`, `obstacle_avoidance_mission`, `gcs_bridge`
Typ:
- `std_msgs/String` JSON, pozdeji `scout_msgs/AvoidanceStatus`
Vyznam:
- runtime stav, progres, blocked, completion

Doporucena pole:
- `cmd_id: str`

8.1 Zakladni pravidlo

Mission vrstva rika:

- kdy je mise hotova

Runtime rika:
- jak se tam lokalne dostat bezpecne
- jestli je target dokoncen
- jestli je doÄŤasne nebo tvrde blokovan

Mission vrstva nesmi:
- posilat raw detour waypointy
- delat left/right avoidance heuristiku

8.3 Doporucene API `status`

Runtime ma reportovat oddelene:
- state
- result
- blocking severity
- planner mode

Priklad:
{
  "cmd_id": "drone_0_cell_x4_y2",
  "route_id": "cell_x4_y2",
  "state": "DETOUR_EXECUTION",
  "result": "ACTIVE",
  "planner_mode": "SUBGOAL_PATH",
  "distance_to_target_m": 6.3,
  "subgoal_ned": [8.0, -2.0],
  "drone_ned": [4.1, -1.2, -5.0],
  "avoidance_active": true,
  "scan_active": false,
  "blocked_severity": "NONE",
  "reassign_recommended": false
}

Pri dokonceni:
- `result = COMPLETED`

Pri blokaci:
- `result = BLOCKED`
- `blocked_severity = SOFT | HARD`
- `blocked_reason = no_local_path | repeated_detour_failure | peer_conflict`
- `reassign_recommended = true/false`

Pri abort:
- `result = ABORTED`

8.4 Jak route provider reaguje

`swarm_agent` / route provider:
- `ACTIVE`: nic nemeni, jen sleduje progres
- `COMPLETED`: posle dalsi cell / waypoint
- `BLOCKED + SOFT`: kratce ceka, target si ponecha
- `BLOCKED + HARD`: oznami coordinatoru `CELL_DEFERRED` nebo `DRONE_BLOCKED`
- `ABORTED`: posle RTH nebo eskaluje operatorovi

8.5 Co je reusable API boundary

Stejne rozhrani musi umet pouzit:
- `obstacle_avoidance_mission`
- budoucĂ­ `swarm_agent`
- potencialne `swarm_center` route provider

To je hlavni duvod, proc target/status maji zustat jednoduche a mission-agnostic.

9. H. SWARM NAPOJENI
======================================================================

9.1 Co musi swarm vedet

Swarm potrebuje vedet jen:
- jestli je dron `AVAILABLE / ACTIVE / BLOCKED / RTH / LANDING / ABORTED`
- jaky route/cell prave dela
- jestli je `reassign_recommended`
- posledni pozici dronu pro peer safety mask

Swarm nepotrebuje vedet:
- lokalni occupancy mapu
- planner internals
- scan yaw logic
- detaily kazdeho lokalniho replanu

9.2 Napojeni do `swarm_agent`

Cilovy stav:
- `swarm_agent` prestane publikovat PX4 setpointy
- bude:
  - prijimat cell assignment
  - prekladat cell center -> `avoidance/target_cmd`
  - sledovat `avoidance/status`
  - reportovat agregovany swarm status

Docasna migrace:
- pridat parametr `navigation_backend := direct | avoidance_runtime`
- default nechat `direct`, dokud avoidance backend nebude stabilni
- po M3 prepnout launch pro Isaac obstacle-enabled misi na `avoidance_runtime`

9.3 Napojeni do `swarm_coordinator`

Ponechat jednoduchost:
- `swarm_coordinator` dal dela task allocation
- navic bude sledovat `DRONE_BLOCKED` a `CELL_DEFERRED`
- pri `HARD_BLOCKED > threshold` muze dron oznacit jako doÄŤasne unavailable

Minimalni nove udalosti do `/swarm/drone_status`:
- `NAV_ACTIVE`
- `NAV_COMPLETED`
- `NAV_BLOCKED_SOFT`
- `NAV_BLOCKED_HARD`
- `CELL_DEFERRED`
- `NAV_ABORTED`

9.4 Napojeni do `home_manager`

Ponechat:
- `home_manager` zustava zdroj `/{drone_ns}/rth_target`

Rozsireni:
- pri `BLOCKED_HARD` muze `swarm_agent` nebo operator rozhodnout o `return_home`
- runtime pak home target normalne zpracuje jako dalsi mission command

9.5 Napojeni do `gcs_bridge`

Pridat forwarding:
- `/{drone_ns}/avoidance/status`
- `/{drone_ns}/avoidance/scan_event`
- volitelne `/{drone_ns}/avoidance/subgoal`
- volitelne `/{drone_ns}/avoidance/local_grid` jen na vyzadani nebo low-rate

Do GCS netlacit:
- plnou mapu 10 Hz
- raw point clouds live

LepĹˇĂ­:
- jen event-driven summaries
- artifacts cist ze sdileneho disku / on-demand

9.6 Temporary blocked drone

Doporucene pravidlo:
- prvnich 15 s je blocked lokalni problem dronu
- po 30 s nebo po 2 scan fail cyklech -> `HARD_BLOCKED`
- coordinator doÄŤasne prestane tomuto dronu pridelovat dalsi bunky
- aktualni bunka se vrati do allocatoru jako `deferred`

9.7 Kdy delat reassign

Reassign smysl dava jen kdyz:
- `blocked_severity = HARD`
- dron je blokovan dele nez `hard_blocked_timeout_s`
- nebo peer conflict trva dlouho

Nedelat reassign:
- pri kazdem kratkem warn nebo jednom scanu

16.1 Priority 1 - core avoidance runtime [DONE]

  - `LOCAL_REPLAN`
  - `DETOUR_EXECUTION`
  - `BLOCKED`
- `scan_complete(success=True)` -> `LOCAL_REPLAN`
- `scan_complete(success=False)` nebo repeated no-path -> `BLOCKED`

Stabilizovany `avoidance/status` kontrakt:
- `phase`, `state`, `result`

Runtime event kontrakt:
- topic `/{drone_ns}/avoidance/events`
- `phase_transition` event
- `scan_complete` event

16.2 Priority 2 - swarm integrace a blocked severity [DONE]

Dokonceno:
- `swarm_agent` ma `navigation_backend=direct|avoidance_runtime`
- v `avoidance_runtime` backendu je interni flight loop swarm agenta vypnuta
  (zadny dual ownership)
- finalni mapping `/{drone}/avoidance/status` -> `/swarm/drone_status`
- `blocked_severity` sjednoceno na `NONE|SOFT|HARD`
- `task_allocator` umi pracovat s novymi swarm status eventy

16.3 Priority 3 - peer/reassign/GCS [DONE - current scope]

Dokonceno:
- `peer_tracks` robustnejsi ingest + smoothing/clamp
- allocator:
  - `TEMP_BLOCKED`
  - deferred queue + metadata
  - HARD blocked -> deferred + cooldown + reassign queue
  - SOFT blocked -> bez forced defer
- `CELL_DEFERRED` semantika zavedena
- `gcs_bridge` forwarduje:
  - `AVOIDANCE_STATUS`
  - `AVOIDANCE_EVENT`
  pres `MSG_DRONE_STATUS` payload

16.4 Priority 4 - typed payloads / cleanup / tuning [DONE - non-breaking]

Dokonceno:
- typed parsery pro avoidance/swarm status payloady
- safe migrace consumeru (`spray_controller`, `cell_data_recorder`)
- legacy `/obstacle_avoidance/*` publikace gate parametrem
  `publish_legacy_obstacle_topics` (default `false`)
- tuning hooks pridany:
  - mapper: `peer_cost_gain`, `blocked_cost_gain`
  - planner: `blocked_cost_scale`, `peer_cost_scale`

16.5 Otevrena rizika po implementaci

- full E2E ROS/PX4 integrace nebyla v tomhle kroku spustena
- GCS detail stream pro avoidance je veden jako subtype uvnitr
  `MSG_DRONE_STATUS` payloadu (consumer musi filtrovat `status` field)
- po prekroceni `max_deferrals_per_cell` je bunka zmrazena v deferred stavu
  bez dedikovaneho global alert topicu
